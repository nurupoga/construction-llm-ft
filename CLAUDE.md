# 建設・不動産ドメインLLMファインチューニング実験（宅建士PoC起点）

> リポジトリ名 `construction-llm-ft` は**建設・不動産業界向けLLM**の包括名として継続使用。
> 初期スコープは宅地建物取引士（宅建士）1試験に確定（2026-04-23方針変更）。
> 過去に施工管理6種ベースで検討していた資料は `docs/archive/` 配下に退避済み。

## 目的
- ファインチューニングを一度通して経験を積む（最優先）
- 建設×AI / 不動産×AI 系の働き口紹介がきっかけ。職務経歴に書ける成果物を作る
- 宅建士はユーザー自身の学習動機とも一致し、知名度と公開時インパクトが高い
- 「学習前 vs 学習後で正答率が上がった」が示せれば十分。論文クオリティは不要

## 作業マシン（メイン: Mac Mini）

- ローカルは議論メモ・タスクリスト閲覧用

## 進捗共有ルール（厳守）
**Mac Mini側Claude**は以下を必ず守る。ローカル新セッションが進捗を取りに来るので、状態が古いとユーザーに混乱を与える。

1. **節目ごとに WORK_LOG.md を更新 → git commit → git push**
   - 節目 = Phaseの1サブタスク完了 / 大きなエラーで方針変更 / ユーザーへの確認事項発生
   - 最低でも1時間に1回はpush（長時間処理中なら「実行中」ステータスをcommit）
2. **進捗状況は `WORK_LOG.md` 末尾に Markdown で書く**
   - 何をやった、結果、次のステップ、ブロッカー、ユーザーへの依頼事項
3. **ユーザー判断が必要になったら `WAITING_FOR_USER.md` を作って push**
   - 待ちが解消されたらこのファイルを削除
   - ローカルセッションがこのファイル存在を確認したら即ユーザーに報告する
4. **作業終了時** は `WORK_LOG.md` の最後に「セッション終了」と明記

**ローカル新セッション側**は以下を守る:
1. 「FT」「建設」「不動産」「宅建」「進捗」関連の話題で起動したら、まず `cd /Users/user/university/construction-llm-ft && git pull` を実行
2. `tail -80 WORK_LOG.md` で最新進捗を確認
3. `WAITING_FOR_USER.md` の存在を確認、あれば内容をユーザーに報告
4. ユーザーへの報告 → 必要ならMac Mini側Claudeへの追加指示を WORK_LOG に書く（or `NEXT_TASK.md` 更新）

## 技術スタック（決定）
- ベースモデル: **Qwen2.5-7B-Instruct**（日本語強い、Colab無料枠と相性◎）
- 手法: **QLoRA(4bit)** + **Unsloth**（T4最適化、素のtransformersの2倍速）
- 形式: instruction-tuning `{instruction, input, output}`
- 学習設定: epoch 2〜3, lr 2e-4, max_seq 2048
- チェックポイント: Google Drive 保存（Colab切断対策）

## データ戦略（確定: 宅建士1試験特化）
- **採用案**: 宅地建物取引士試験の過去問特化
- **理由**:
  - 年1回50問4択、問題構造がクリーンで整形コストが低い
  - ユーザー自身の学習動機と一致（副次価値あり）
  - 知名度が高く、公開時の読者・採用担当者への訴求力大
  - 施工管理系と比べ利用規約面も相対的にクリーン（要検証、Phase 2-3で確認）
- **試験種スコープ（確定 2026-04-23 方針変更）**:
  - **PoC対象**: 宅地建物取引士試験（宅建士）1種
  - **データ量**: 直近10年分（2016〜2025）= 500問オーダー（50問/年）
  - 主催団体: 一般財団法人 不動産適正取引推進機構 (RETIO)
  - 試験コード（内部管理用）: `takken`
- **学習/評価分割**（時系列holdout）:
  - **train**: 2016〜2024 年度（9年分、約450問）
  - **eval**: 2025 年度（1年分、50問）
  - 2020年は10月/12月の分割実施 → 両方train側に含める想定（Phase 2-1の調査結果で再検討）
- **解説テキストの扱い**: 出版社解説は著作権回避のため使用不可。自作要約 or **Gemini API による合成解説**（Phase 3）で作る
- **以前の検討（施工管理6種）**: `docs/archive/data-sources-construction.md` と `docs/archive/license-check-construction.md` に退避。将来の拡張候補として参照可能

## 評価戦略
- **必須**: FT前ベースライン取得（取らないと効果判定不能）
- 同一スクリプト・同一プロンプト形式・同一採点ロジックで前後比較
- 出力: 学習前後の正答率を1枚のグラフに
- （オプション）JMMLU 100問抜粋で汎用能力低下の軽い目視確認

## やらないこと（過剰設計回避）
- catastrophic forgetting の厳密検出
- 統計的有意差の検証
- 複数シードでの安定性確認
- 評価QAの自作（過去問の別年度で十分）

## ハマりどころ（事前認識）
1. データ整形・再現性管理が真の鬼門（Unsloth以前の問題）
2. train/eval混入に注意（2025年度50問は絶対に学習に混ぜない）
3. 学習年と評価年が近すぎると比較が崩れる → 2024→2025は時系列順守
4. Colab セッション切断 → チェックポイントをDriveに必ず保存
5. プロンプト形式・問題数・採点法を前後で固定（`docs/freeze-spec.md`）
6. 宅建士は法改正（宅建業法・民法等）があるので、学習期間中に改正が絡む問題は解答の妥当性が変わる可能性 → 評価年度の出題法令基準日を明記
